# 0001: 文档策略与核心约束

**状态**：接受

**日期**：2026-06-14

**决策者**：通过 grill-me 过程与用户共同确认

## 背景

项目文档长期缺乏系统整理，包含大量过时信息和 AI 生成的规划内容。代码库实际能力已演进（新增 udp、shm、process_trade 等交易专用监控器，完整的 Prometheus + Grafana + 告警集成），但文档严重滞后且混杂“下一阶段工作规划”、“长期愿景”等无法验证的内容。

本次通过结构化 grilling 过程，对文档重构、项目身份、硬约束和工作优先级达成一致。

## 决策

1. **文档卫生规则**  
   - 永远不撰写“未来愿景”、“长期规划”、“业界最佳实践启示”、“云原生转型储备”等内容。  
   - 所有内容必须是**当前真实事实**或**有明确验收标准的短期可验证目标**。  
   - 历史规划类内容一律删除，不归档。

2. **采用 ADR 机制**  
   关键策略、架构约束和重大决策必须通过 ADR 记录，作为不可变的历史记录。

3. **单一活文档**  
   建立 `ROADMAP.md` 作为下一阶段工作的**唯一真相来源**。所有规划相关内容只允许出现在 ROADMAP 中，且必须按优先级和依赖排序。

4. **硬约束：不迁移 CO-RE**  
   持续使用 BCC + kprobe/tracepoint 路线，保持对老内核（3.10+）的兼容性。任何新监控器必须在此约束下务实实现。

5. **项目真实身份**  
   本项目当前是**eBPF 监控系统**，重点服务特定进程。存在高度领域化的专用监控器（udp、shm、特定进程），而非通用全栈监控工具。

6. **低延时网卡监控原则**（中科驭数网卡）  
   - 作为独立监控器实现。  
   - 核心关注：延迟信息、硬件队列深度、缓冲区数据。  
   - 实现手段：tracepoint/kprobe 务实选择（具体型号待补充）。  
   - 范围：系统级全量监控。  
   - 输出模式（聚合 vs 事件）：必须先通过验证 spike 确定数据量后再决策。

7. **工作执行顺序**（已锁定）  
   1. 文档重写 + 债务补写 + 第一个 ADR  
   2. 低延时网卡验证 spike  
   4. 基础测试框架建设（Python 侧优先）  
   5. 完善数据分析能力 + 完善数据展示能力  
   3. 正式低延时网卡监控实现

8. **具体文档操作**  
   - 老规划类文档直接删除。  
   - `docs/PROMETHEUS_GRAFANA_GUIDE.md` 内容合并进 `USER_GUIDE.md`。  
   - 必须完成历史文档债务补写（udp、shm、process_trade、Prometheus 集成、context_switch 等实际能力）。

9. **未来变更要求**  
   任何涉及策略、约束或重大能力变更，必须更新 ROADMAP 或新增 ADR。

## 结果

- 重写 `README.md`、`docs/ARCHITECTURE.md`、`docs/USER_GUIDE.md`（含 Prometheus 合并）。
- 新增 `docs/ROADMAP.md` 和 `docs/adr/` 目录。
- 删除 analysis/reports/ 下所有老规划文档。
- 后续所有工作严格遵循以上规则。
